# The Sadgi Receipt Standard (SIP-1)

A **Sadgi Receipt** is the core cryptographic primitive of the protocol. It is a standardized byte array that proves a specific zkVM execution occurred correctly and produced a specific public output (the Journal).

## Motivation
For Soroban smart contracts to trustlessly consume off-chain computations, they need a deterministic, easily parsable data structure. If every application invents its own receipt format, interoperability between DeFi, KYC, and Enterprise applications is impossible.

## The Data Schema

A valid `ProofReceipt` MUST be serialized using Soroban's native XDR types to minimize gas costs during on-chain verification.

```rust
use soroban_sdk::{contracttype, Bytes, BytesN};

#[contracttype]
pub struct ProofReceipt {
    /// The 32-byte SHA-256 identifier of the zkVM Guest Program (e.g. SP1 Image ID).
    pub image_id: BytesN<32>,
    
    /// The 32-byte Soroban Contract ID that requested this proof. 
    /// This strictly prevents Replay Attacks across different contracts.
    pub caller_contract_id: BytesN<32>,
    
    /// The public outputs emitted by the zkVM program.
    pub journal: Bytes,
    
    /// The cryptographic seal/proof generated by the Prover.
    pub seal: Bytes,
}
```

## Replay Attack Protection
To prevent a malicious actor from intercepting a valid receipt and submitting it to a *different* contract to extract value, the `caller_contract_id` is cryptographically committed into the `journal` by the zkVM program. The Soroban Verifier MUST assert that `caller_contract_id == env.caller()`.

## Verification Flow
When a Soroban contract receives a `ProofReceipt`, it makes a Cross-Contract Call to the Official Sadgi Verifier Contract:

```rust
let is_valid = sadgi_verifier_client.verify(&receipt);
assert!(is_valid, "Invalid Zero-Knowledge Proof");
```
